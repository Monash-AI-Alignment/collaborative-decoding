Weak-to-Strong Collaborative Decoding for Fingerprint Removal

Autonomous agents search for the deferral policy that lets a small, un-fingerprinted model write as much of the answer as possible while matching a large model's quality — and we ask whether the large model's watermark survives the dilution.

A model fingerprint or watermark lets a provider prove that some text came from their model — to enforce terms of use, detect model theft, or trace a leaked checkpoint. WatermarksWe use "watermark" for a detectable signal added to outputs and "fingerprint" for one baked into weights; both are detected the same way here. perturb generation so outputs carry a statistically detectable signal invisible to a reader; the canonical scheme is the Kirchenbauer et al. red-green watermark . Fingerprints bake an owner-specific, semantically triggered signal into the weights . Both are only as useful as they are robust: if a cheap post-processing step removes the signal while preserving the text's usefulness, the guarantee is hollow.

This report studies one such step, deliberately cheap: no fine-tuning, no gradients on the target, no detector queries — just inference-time orchestration of a small model the attacker owns alongside the watermarked target. EDIT: rewrite this motivation in your own voice.

Weak-to-strong collaborative decoding

Fix a cheap weak model (here Llama-3.2-1B, white-box — we can read its per-token probabilities) and an expensive strong model (Qwen2.5-7B, treated as black-box — text in, text out). We generate each answer collaboratively: the weak model proposes tokens greedily, and at every step a deferral policy decides whether to keep the weak model's token (CONTINUE) or hand the next span to the strong model (DEFER). Because the two models tokenize differently, every handoff is at the text level, not the token level.Spans are composed as strings and re-tokenized on each side, keeping the collaboration tokenizer-agnostic.

Two quantities score a policy. The first is how much work the weak model did: $f_\text{weak}$, the fraction of output characters produced by the weak model. The second is whether quality held up, measured against the strong model itself. With $U$ the method's task utility (an LLM-judge win-rate on AlpacaEval against the strong model's own outputs, so the strong model scores $U_\text{strong}=0.5$ against itself), define utility recovery:

\text{recovery} \;=\; \frac{U - U_\text{weak}}{U_\text{strong} - U_\text{weak}}

$\text{recovery}=1$ means the collaboration is as good as the strong model; $\text{recovery}=0$ means no better than the weak model alone. The objective is to

\text{maximize}\; f_\text{weak} \quad \text{subject to} \quad \text{recovery} \ge 0.98 .

Push as much generation as possible onto the cheap model without losing quality. This is a genuinely hard online search — the policy sees only white-box scalars from the weak model (entropy, top-1 probability, top-2 margin, the candidate token's identity, and the text so far) and must decide, per token, when the weak model is about to say something it will get wrong.

Autonomous agents as the researchers

Rather than hand-design the policy, we let autonomous Claude agents discover it, using Anthropic's Automated Weak-to-Strong Researcher framework for AI-driven research .Two unrelated senses of "weak-to-strong" meet here: the framework's name refers to the alignment problem of supervising a strong model with weak signals, whereas our task is weak-to-strong collaborative decoding. We borrow the autonomous-researcher machinery, not the alignment setup. Each agent is a fresh session on a GPU node with the decoding engine, the judge, and tools onto a shared forum: read the leaderboard, read all findings, evaluate a candidate, post a result. An agent reads what others found, proposes a policy, writes its code, evaluates against a fixed canonical reference, and publishes what it learned — including refutations of its own hypotheses. The orchestrator recomputes recovery server-side and $f_\text{weak}$ is engine-measured, so the leaderboard cannot be set by a self-reported number. The live forum, leaderboard, and every measured policy — 24 findings across 10 policy families so far — are on the dashboard; this page is a frozen narrative over that output.

What they found: seamgate

The best certified policy is seamgate, and its lineage is instructive. An earlier policy, margin-threshold, deferred whenever the weak model's top-two probability margin fell below a threshold — whenever it was unsure. That over-defers: a 1B model is unsure about many harmless choices (which synonym, which connective) the judge does not care about. The content-gate insight was to defer on uncertainty only when the uncertain token is content-bearing — a number, a proper noun, or a long word — and let the weak model carry its own phrasing wobble. Seamgate adds one refinement: on a defer, the strong model writes to a sentence boundary before handing back, so the weak model resumes at a clean seam.

Seamgate, precisely. On each weak-model token, defer iff the top-1-minus-top-2 probability margin is below $\tau = 0.16$ and the greedy token is content-bearing (contains a digit; or is a mid-sentence capitalized token; or has $\ge 4$ letters). On a defer, the strong model generates up to $\text{span\_max}=64$ tokens, stopping at the first sentence boundary.

Against the canonical $n=100$ reference, seamgate reaches $f_\text{weak}=0.466$ at $\text{recovery}=1.047$ — the weak model writes ~47% of the characters at quality indistinguishable from (slightly above) the strong model. That is the operating point we carry into the attack.

The fingerprint-removal reframe

Nothing above mentions watermarks — the agents optimize only quality-preserving cheapness. Now reinterpret the same machinery as an attack. Let the strong model be a watermarked target queried as a black box, and the weak model an un-watermarked model the attacker owns. Then $f_\text{weak}$ is exactly the fraction of the final text that did not come from the watermarked model — the dilution. A high-$f_\text{weak}$ policy that preserves quality is, from this angle, a cheap laundering scheme: most of the text is the attacker's model, the target is invoked only for the hard spans, and the hope is that the watermark — carried by a minority of tokens — falls below detection.

The watermark p-value is never an optimization target. The agents optimize $f_\text{weak}$ and recovery only; they never see the detector. We measure detection post-hoc on the utility-optimal policy. So this is not a policy tuned to evade a detector, but a quality-driven policy that happens to dilute.

Two watermarks, one detector family

We test two watermarks so the result does not hinge on one installation. Both are KGW-family — they partition the vocabulary into a pseudo-random green list (seeded by the previous token) and detect an excess of green tokens with a z-test — but they are installed very differently.

WatermarkHow it is installedStrong modelDomain
Semantic KGW fingerprintFine-tuned into the weights to activate on one domain Qwen2.5-3B, fine-tunedFrench
Red-green KGW (the original)Inference-time logit bias, no fine-tuning Qwen2.5-7B + logit processorEnglish

The inference-time watermark adds a bias $\delta=2$ to the green-list logits each step (green fraction $\gamma=0.25$); the detector is the standard green-count z-test. Generator and detector are built from the same green-list construction, so they agree by design. A batch is judged watermarked when the detector p-value falls below $\alpha=10^{-3}$.Following the fingerprinting detector's convention; $p < 10^{-3}$ ⇒ "watermarked". Two controls anchor every run: strong-only (the pure watermarked model) must read as watermarked, and weak-only (the pure un-watermarked model) must read as clean.

Does the fingerprint survive?

It does — under both watermarks, with the weak model writing roughly half the text.

The shape is the same in both columns. Detection is overwhelming ($p\approx0$, far below $10^{-3}$) at $f_\text{weak}=0$ and still overwhelming at $f_\text{weak}\approx0.5$. The mark only escapes detection at $f_\text{weak}=1$ — pure weak-only — where by construction none of the text came from the watermarked model. The three points we have are consistent with a flat-then-cliff dilution curve — robust until the strong model is essentially removed — though we have not traced the shape between $f_\text{weak}\approx0.5$ and $1$ (see Limitations).

A second observation, in the English column: the inference-time watermark also dragged seamgate's recovery from ~1.05 down to 0.77. This is a single-run comparison — larger than the agents' documented ~0.06 recovery noise at $n=100$, but not isolated against a matched watermarked strong-only baseline — so read it as a hypothesis: the green-list bias appears to erode the strong model's own spans, the very spans seamgate leans on for the hard content it defers. If so, the watermark is doubly inconvenient for an attacker: hard to dilute and it erodes the quality the attacker is trying to keep.

Why it holds for both installations

That the same picture holds for a fine-tuned fingerprint and an inference-time watermark, in two languages, suggests the cause is structural rather than a quirk of one installation. The argument below is specific to per-token green-list watermarks (both schemes we tested); a sentence-level or semantic-embedding watermark could dilute differently, so this is not a claim about watermarks in general (see Limitations).

The argument is short. A per-token green-list watermark places its signal on the strong model's tokens. To preserve the strong model's utility, a quality-constrained policy must keep enough strong-model tokens — precisely on the content-bearing spans where the weak model would otherwise err. Those are the tokens that carry the mark. So the two things the attacker wants — keep the quality and drop the watermark — pull on the same lever in opposite directions. You cannot dilute below detection without deferring so little to the strong model that quality collapses.

Takeaway. Weak-to-strong collaborative decoding is a real efficiency win — a 1B model can carry ~half of a 7B model's answers at parity. But as a fingerprint-removal attack it fails against KGW-family watermarks: any operating point that preserves utility also preserves detectability. This is a positive result for watermark robustness. EDIT: state the defensive implication the way you want it framed.

Limitations

  • One watermark family. Both watermarks are KGW-style green-list schemes with the same per-token z-test detector. A watermark that spreads its signal differently (sentence-level, semantic-embedding) might dilute differently; the structural argument above does not cover it.
  • Few operating points. We measure three points per watermark ($f_\text{weak}\in\{0,\,{\sim}0.5,\,1\}$); we have not swept $\tau$ to trace where between 0.5 and 1 detection breaks.
  • Detector subsampling. The detector's green-list mask is large; on the longest completions it was subsampled (as few as 25 of 100 concatenated completions) to fit in memory. Detection was still $p\approx0$, and more completions only strengthen it, so this does not affect the conclusion.
  • Sub-parity attack points, and a French confound. On the watermarked setups seamgate sits below the $\text{recovery}\ge0.98$ bar (0.77 English, 0.50 French). The French number reflects more than domain non-transfer: the French target is a different, smaller base model (Qwen2.5-3B fine-tuned) run with thresholds ($\tau$, $\text{span\_max}$) tuned on the 7B English setup — a model-and-threshold mismatch. Either way, a re-tuned parity point would defer more to the strong model, i.e. carry more watermarked tokens, so the survival conclusion only strengthens.
  • Scale. $n=100$ prompts per setting.
